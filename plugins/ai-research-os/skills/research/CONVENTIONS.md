# Research Directory — Conventions

The data contract for research directories. The `/research`, `/research-lint`, `/research-distill`, and `/research-render` skills all read and write against this layout. If you're operating on a research dir, this file is authoritative — when a SKILL.md and CONVENTIONS.md disagree, CONVENTIONS.md wins.

This file describes **structure, naming, schemas, and invariants**. Operations (how to ingest, how to query, how to lint) live in the individual SKILL.md files.

---

## Purpose

A research directory is a personal LLM-maintained knowledge base for one topic. It compounds over time: every ingest adds raw sources AND updates synthesis; every saved query becomes a wiki page; every lint pass keeps it consistent.

Three layers, in priority order for any agent reading the dir:

1. **`index.md` / `index.yaml`** — the catalog. Always read first.
2. **`wiki/`** — LLM-maintained synthesis. Read this before raw.
3. **`raw/`** — immutable copies of source documents (plus original PDFs and images in `raw/assets/`). Read only when wiki + index are insufficient.

---

## Directory layout

```
<research_dir>/                     # e.g. working-dir/research-<topic-slug>/
├── README.md                       # 5-line autogenerated signpost
├── index.yaml                      # canonical, machine-readable
├── index.md                        # AUTOGENERATED from index.yaml
├── log.md                          # append-only, ## [YYYY-MM-DD] <op> | <subject>
├── raw/                            # immutable source copies
│   ├── <source-slug>.md
│   ├── readwise-<slug>.md
│   ├── nlm-<slug>.md
│   └── assets/                     # downloaded images + original PDFs
│       └── <source-slug>/
│           ├── img-001.png
│           ├── img-002.jpg
│           └── original.pdf
└── wiki/                           # LLM-maintained synthesis (mutable)
    ├── overview.md                 # lay of the land — what's in this research
    ├── synthesis.md                # the thesis — what it all means
    ├── sources/<slug>.md           # per-source extended summary + cross-refs
    ├── entities/<name>.md          # person / tool / framework
    ├── concepts/<name>.md          # idea / pattern
    ├── comparisons/<slug>.md       # X-vs-Y synthesis pages
    ├── questions/YYYY-MM-DD-<slug>.md   # resolved Q&A from /research
    ├── open-questions.md           # gaps / suggested investigations (rolling)
    ├── contradictions.md           # claims that disagree (rolling)
    ├── repos/<repo>/               # GitHub two-tier
    │   ├── ARCHITECTURE.md
    │   └── <module>.md
    └── renders/                    # multi-form answers, filed back into wiki
        ├── marp/<slug>.md
        ├── charts/<slug>.png       # + companion <slug>.py
        └── canvases/<slug>.canvas
```

**Layout note (v4):** `index.yaml`, `index.md`, `log.md`, `raw/`, and `wiki/` all sit directly under `<research_dir>`. Earlier versions wrapped them in a `memory/` subdirectory; that wrapper is gone. Older dirs must be flattened into v4 before they can be appended to.

**Invariants:**
- `raw/` is **immutable**. Files land here on ingest and are never edited. Re-ingesting an existing source overwrites only with explicit user consent.
- `raw/assets/` is also immutable — downloaded images and original PDFs.
- `wiki/` is **mutable** and owned by the LLM. Humans read it; humans edit it only as a deliberate override.
- `index.yaml` is **canonical**. `index.md` is **always regenerated** from it.
- `log.md` is **append-only**. Never rewrite history.

## File naming

- **Slugify** all derived filenames: lowercase, kebab-case, ASCII only, strip punctuation, max 60 chars.
- **Readwise** raw files: prefix with `readwise-` (e.g., `raw/readwise-anatomy-of-an-agent-harness.md`).
- **NotebookLM** raw files: prefix with `nlm-` (e.g., `raw/nlm-context-engineering-talk.md`).
- **YouTube** raw files: prefix with `youtube-` (e.g., `raw/youtube-agent-harness-demo.md`). The raw file is caption-transcript markdown with timestamped snippets; it is treated as the complete Layer 3 source.
- **Web seeds**: no prefix; slug from page title.
- **Obsidian notes**: no prefix; slug from note title.
- **GitHub**: stored ONLY under `wiki/repos/<repo>/` (curated `ARCHITECTURE.md` + per-module `<module>.md`). The actual repo source code lives outside the research dir in a `.github-cache/` placed as a sibling of it (in the research dir's parent — a reusable shallow clone, ephemeral). Do not slugify repo or module names — preserve them as-is for traceability. **GitHub is the one source type that has no `raw/` entry** — its "raw" is upstream code, pinned by `github_commit_sha`.
- **Wiki entity/concept pages**: slug from canonical name. Aliases handled via frontmatter, not filename.
- **Wiki source pages**: filename matches the raw file slug (e.g., `wiki/sources/readwise-anatomy-of-an-agent-harness.md`).
- **Question pages**: `YYYY-MM-DD-<short-question-slug>.md`. Date is the day the question was asked.
- **Comparison pages**: `<a>-vs-<b>.md`, both slugs alphabetized (e.g., `comparisons/bm25-vs-hybrid-retrieval.md`).
- **Render pages**: format-specific dir + slug derived from the source wiki page or query (e.g., `renders/marp/agent-loops.md`).

## index.yaml schema

The full schema lives in `scripts/build_index_yaml.py`. High-level shape:

```yaml
topic: <string>
input_summary: <string>
created: <ISO-8601>
last_updated: <ISO-8601>
total_sources: <int>
total_wiki_pages: <int>
sources:
  - title: <string>
    origin: obsidian | readwise | web | notebooklm | github | pdf | youtube
    original_path: <vault path | URL | nlm:// | github:// | pdf:// | youtube://>
    source_url: <URL or null>
    authors: [<string>, ...]
    published_date: <YYYY-MM-DD or null>
    publication: <string>
    relevance_score: <float 0.0–1.0>     # 1.0 = seed (user-provided); 0.8 = high / 0.5 = medium (from the researcher relevance tag)
    summary: <string>
    tags: [<string>, ...]
    uri_full: raw/<filename>              # Layer 3
    uri_highlights: raw/<filename> | null # Layer 2 (only when user-curated)
    uri_source_page: wiki/sources/<filename>   # per-source wiki page (Phase 3+)
    assets: [raw/assets/<slug>/...]       # downloaded images + original PDF (when present)
    text_quality: high | low              # only set on PDFs lacking proper OCR
    # origin-specific fields:
    readwise_location: library | feed     # readwise only
    nlm_source_id: <uuid>                 # notebooklm only
    nlm_notebook_title: <string>          # notebooklm only
    nlm_content_type: note | source       # notebooklm only
    github_repo_url: <URL>                # github only
    github_commit_sha: <sha>              # github only
    github_branch: <string>               # github only
    github_files: [<path>, ...]           # github only
    youtube_video_id: <string>            # youtube only
    youtube_url: <URL>                    # youtube only
    youtube_channel: <string or null>     # youtube only
    duration_seconds: <int or null>       # youtube only
    transcript_source: transcript_api | manual  # youtube only
    transcript_language: <string or null> # youtube only
    transcript_language_code: <string or null> # youtube only
    transcript_is_generated: <bool or null> # youtube only
    timestamps_available: <bool>          # youtube only
    wiki_refs: [<wiki-page-path>, ...]    # which wiki pages cite this source
ingest_log:
  - date: <YYYY-MM-DD>
    sources_added: <int>
    rounds: <int or null>                 # 0 for append (seed-only ingest); 1/2/3 for deep (fast/light/deep)
```

**Sort order in index.yaml**: seeds (score 1.0) first, then by `relevance_score` descending. Handled deterministically by `build_index_yaml.py` — never sort by hand.

## index.md autogeneration

`index.md` is **never hand-edited**. It is regenerated from `index.yaml` at the end of every ingest, lint, search-with-save, and render operation that mutates the index.

Required header:

```markdown
<!-- AUTOGENERATED from index.yaml. Do not edit. Run /research, /research-lint, or scripts/build_index_md.py to regenerate. -->
```

Conventions:
- A "Wiki pages" section at the top listing `wiki/overview.md`, `wiki/synthesis.md`, then sections for entities, concepts, comparisons, questions, plus `wiki/open-questions.md`, `wiki/contradictions.md`, `wiki/repos/`, `wiki/renders/` — all rendered as Obsidian `[[wikilinks]]`.
- One section per source category (seeds, then by origin).
- Each source rendered as a bullet with a wikilink to the raw file: `- [[raw/<slug>|Title]] (score: 0.92, readwise) — summary first sentence.`
- Sections are stable and ordered alphabetically within each category.
- Output is deterministic: identical `index.yaml` → byte-identical `index.md`.

## log.md format

Append-only. **Every line that starts a new entry must use this exact prefix**:

```markdown
## [YYYY-MM-DD] <op> | <subject>
```

Where `<op>` ∈ `ingest`, `query`, `lint`, `distill`, `render`, `migrate`. This makes the log greppable with no tooling: `grep "^## \[" log.md | tail -10`.

Body of each entry: 2–10 lines. What changed, what sources, what wiki pages updated, what was decided. No prose narrative — bullets or short sentences.

## Wiki page types (the full set)

| Type | Path | Purpose | Update trigger |
|---|---|---|---|
| `overview` | `wiki/overview.md` | Lay of the land — what's in this research, organized | Every ingest |
| `synthesis` | `wiki/synthesis.md` | The thesis — what it all means, ≤ 1000 words | Every ingest |
| `source` | `wiki/sources/<slug>.md` | Per-source extended summary, key claims, notable quotes, connections | Every ingest, one per source; revised when newer sources reframe it |
| `entity` | `wiki/entities/<slug>.md` | Person / tool / framework appearing in ≥2 sources | Threshold crossed OR new claim added |
| `concept` | `wiki/concepts/<slug>.md` | Idea / pattern appearing in ≥2 sources | Threshold crossed OR new claim added |
| `comparison` | `wiki/comparisons/<slug>.md` | X-vs-Y synthesis | User asks OR lint detects mutual reference |
| `question` | `wiki/questions/YYYY-MM-DD-<slug>.md` | Resolved Q&A from `/research` | User bookmark OR auto-save (≥2 sources cited) |
| `open_question` | `wiki/open-questions.md` (rolling) | Gaps / suggested investigations | Lint output OR user-flagged from query |
| `contradiction` | `wiki/contradictions.md` (rolling) | Source disagreements | Lint detects |
| `repo` | `wiki/repos/<repo>/ARCHITECTURE.md` | GitHub repo wiki hub | GitHub ingest |
| `module` | `wiki/repos/<repo>/<module>.md` | Per-module spec | GitHub ingest |
| `render` | `wiki/renders/<format>/<slug>.{md,png,canvas}` | Multi-form answer (Marp / chart / canvas / table) | `/research-render` invocation |

## Wiki page frontmatter

Every wiki page has YAML frontmatter:

```yaml
---
type: overview | synthesis | source | entity | concept | comparison | question | open_question | contradiction | repo | module | render
name: <canonical name>
aliases: [<string>, ...]               # optional, for entity/concept
sources: [<original_path>, ...]        # which raw sources back this page
related: [<wiki-page-path>, ...]       # cross-refs to other wiki pages
created: <ISO-8601>
last_updated: <ISO-8601>
---
```

**Source-page frontmatter (extended):**
```yaml
---
type: source
title: <full source title>
original_path: <as in index.yaml>
raw_file: raw/<slug>.md
assets: [raw/assets/<slug>/img-001.png, ...]
authors: [...]
published_date: <YYYY-MM-DD or null>
relevance_score: <float>
ingested: <ISO-8601>
last_updated: <ISO-8601>
entities: [[wiki/entities/...], ...]
concepts: [[wiki/concepts/...], ...]
text_quality: high | low               # PDFs only
---
```

**Render-page frontmatter:**
```yaml
---
type: render
format: marp | chart | canvas | table
sources: [<wiki-page-paths>]
prompt: <the question / spec that produced this render>
created: <ISO-8601>
---
```

## Linking conventions

- Wiki ↔ wiki: Obsidian `[[wikilinks]]` (e.g., `[[wiki/concepts/agent-loop]]`).
- Wiki → raw: also `[[wikilinks]]` to the raw file (e.g., `[[raw/readwise-anatomy-of-agent-harness]]`).
- Wiki → external: standard markdown links.
- Use **wikilinks everywhere inside the research dir** so Obsidian's graph view lights up.

**Citation rule**: every claim in a wiki page that comes from a specific source must link to either the raw file or the per-source wiki page by wikilink. No floating claims. If a claim is the LLM's synthesis, mark it explicitly: `> Synthesis: ...`.

## Source URI conventions

`original_path` in `index.yaml` uses these prefixes:

- Vault notes: relative path from vault root (e.g., `Notes/Some Note.md`)
- Web URLs: full `https://...`
- Readwise: the Readwise document URL or `readwise://document/<id>`
- NotebookLM: `nlm://note/<id>` or `nlm://source/<id>`
- GitHub: `github://<owner>/<repo>@<commit_sha>`
- User-dropped PDFs: `pdf://<original-filename>` (the PDF itself is preserved at `raw/assets/<slug>/original.pdf`)
- YouTube: `youtube://<video_id>` with `youtube_url` preserving the original URL

These are stable identifiers. Use them for deduplication and for matching in `/research-distill`.

## Three-layer progressive disclosure

For any source, an agent reads in this order and stops as soon as the question is answered:

1. **Layer 1 — `summary`** in `index.yaml`. Always available. ~2-3 sentences.
2. **Layer 2 — `uri_highlights`**. Optional. Exists only when the source carries **manually user-curated** highlights (typically Readwise-synced). Never synthesized by an LLM.
3. **Layer 3 — `uri_full`**. Usually available. The complete document.

A fourth, intermediate option exists once the wiki layer is built:

- **Layer 1.5 — `uri_source_page`** (`wiki/sources/<slug>.md`). LLM-extended summary that's denser than the index summary but lighter than the full document. Use this between Layer 1 and Layer 2/3 — read it instead of the full document when wiki integration is enough.

Do not skip Layer 1. Do not bulk-read Layer 3.

## Web content fetching

Generic `http(s)://` sources — anything that is **not** a vault path, Readwise reference, GitHub repo, NotebookLM URI, YouTube URL, or `.pdf` — are fetched with **`curl`** and reduced to readable text by a stdlib-only `python3` HTML stripper (no extra dependency; canonical recipe in `SKILL.md` Step 1, point 3). This works on normal, server-rendered HTML sites only — it does **not** clear bot-blocking, CAPTCHAs, paywalls, JS-only rendering, or geo-walls, and there are no platform-specific extractors. **WebFetch is the fallback** — used when `curl` exits non-zero or returns suspiciously little text (< ~500 chars: a JS-only shell or bot wall); runs that fall back should say so. This applies wherever the family fetches a remote page: web seeds (orchestrator Step 1) and the Readwise→URL fallback in the builder. The fetch tool used is recorded in `log.md`.

## YouTube video handling

Public YouTube URLs are first-class seed sources and should be routed before generic web scraping. They are extracted with public YouTube captions via `scripts/youtube_extract_transcript.py`, which writes timestamped transcript markdown to `raw/youtube-<slug>.md` without requiring an API key.

- YouTube sources use `origin: youtube`.
- `uri_full` points at the generated raw markdown file.
- `uri_highlights` is always `null` unless the user separately provides manual highlights.
- The raw markdown should preserve timestamps, spoken claims, mentioned tools/repos/papers, notable quotes, and open questions visible in the captions. It should not invent visual observations that are not present in the transcript.
- The source page should preserve timestamp anchors wherever possible.
- Requires public captions/transcripts to be available for the video. If captions are unavailable or blocked, skip YouTube ingestion with a clear warning unless a manual transcript is provided.

## Image and PDF handling

**Images.** During ingest, every image referenced in a raw source is downloaded to `raw/assets/<source-slug>/`. The raw markdown copy is rewritten so image references point at the local path. The wiki source page lists assets in frontmatter.

**Two-pass methodology.** Read text first; then load specific images that matter for the entity/concept/source page being written. Not all images are viewed — only those the writer references.

**PDFs.**
- User-dropped PDFs: extract to markdown via `pymupdf4llm`. Markdown becomes the raw file; original PDF preserved at `raw/assets/<slug>/original.pdf`.
- Web PDFs (URL points at `.pdf`): download → extract → same flow; if the direct download is bot-blocked, retry the download with `curl` (`curl -fsSL -A "<browser UA>" … -o original.pdf`).
- Readwise PDFs: already extracted to markdown by Readwise CLI; no special handling.
- NotebookLM PDFs: already text via `nlm source content`; no special handling.
- Image-rich PDFs (slide decks, papers): images are extracted alongside text, treated identically to inline images.
- Scanned PDFs without OCR: flag in the source page frontmatter (`text_quality: low`).

The extraction tool + version is recorded in `log.md` for reproducibility.

## Operation-to-side-effect map

What each operation is allowed to write:

| Operation | raw/ | raw/assets/ | wiki/ | index.yaml | index.md | log.md | Vault |
|---|---|---|---|---|---|---|---|
| `/research` (init) | create | create | create | create | create | create | — |
| `/research` (append) | add files | add assets | add/update pages | update | regenerate | append | — |
| `/research` (query, no save) | — | — | — | — | — | append (optional) | — |
| `/research` (query, save Q&A) | — | — | add `wiki/questions/...` | update counts | regenerate | append | — |
| `/research-lint` | — | — | edit/create/delete | update | regenerate | append | — |
| `/research-distill` | — | — | — | — | — | append | write `research.md` next to content |
| `/research-render` | — | — | add `wiki/renders/...` | update counts | regenerate | append | — |

`/research-distill` is read-only on the research dir except for a log entry.

## Idempotency and safety

- Re-running `/research` with the same inputs must produce no diffs (other than `last_updated` and a log entry).
- Re-running `index.md` regeneration must produce byte-identical output for identical `index.yaml`.
- `/research-lint` must be safe to run repeatedly — it edits but does not duplicate.
- `/research-render` re-rendering with same input is idempotent.
- Never delete from `raw/` or `raw/assets/` automatically. Lint may flag orphans; deletion is a deliberate user action.

---

## What NOT to put in a research dir

- Original conversation transcripts that produced the research (those live in chat history).
- Generated content drafts (those live in the project dir).
- Skill source files or tooling (those live in the plugin).
- Anything from `working-dir` that is not a research artifact.
